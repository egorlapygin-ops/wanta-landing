## Что построено

### Блок 1 — Парсинг описаний из Instagram/TikTok
- yt-dlp извлекает описание и автора из Reels/TikTok без скачивания видео
- Cookies из Chrome для авторизации Instagram
- instaloader как fallback для фото/карусельных постов
- **Результат: 9/11 URL парсятся**

### Блок 2 — Fast Path: описание → артикулы → ссылки
- Regex извлекает артикулы WB/Ozon из текста (форматы: #, арт, wb:, числа 6-10 цифр)
- Ссылки: WB, Ozon, Яндекс Маркет
- Опциональное обогащение через WB CDN API (название, бренд, категория, фото)
- **Результат: 100% точность, 100% покрытие**

### Full Path — OCR видео + транскрибация аудио
- Видео скачивается во temp, удаляется после обработки (<60 сек)
- Извлечение кадров: каждые 0.3 сек + scene detection (порог 0.15)
- **Адаптивная дедупликация**: ≤15с — без дедупа (каждый кадр ценен), >15с — дедуп 3% (экономия токенов)
- max_frames: 80 (было 30)
- OCR кадров: GPT-4o Vision, HIGH quality, батчи по 10 с перекрытием 2, двухпроходная верификация
- **Промпт v2** (A/B тестирован на 23 рилсах): 0 фантомов, 97% точность, визуальное определение маркетплейса
- Артикулы 6-12 цифр (ЯМ использует длинные артикулы)
- Транскрибация: Whisper API → regex на транскрипте
- Карусели: instaloader скачивает фото → тот же OCR
- OCR retry: при сбое батча — повтор через 3 секунды
- **Результат: 76/79 товаров (96% recall), 97% точность, ~$0.05/рилс**

### Мультимаркетплейс
- Wildberries: `wildberries.ru/catalog/{article}/detail.aspx`
- Ozon: `ozon.ru/product/{article}`
- Яндекс Маркет: `market.yandex.ru/card/x/{article}` (прямая карточка товара)
- Автоопределение маркетплейса из OCR (по логотипу/тексту на видео)
- Для ЯМ-товаров артикул отображается в тексте (для ручного поиска)
- **Результат: 15/15 товаров, все три маркетплейса определены корректно**

### Блок 3 — Telegram-бот
- aiogram 3, long polling
- Пользователь шерит ссылку → бот отвечает карточкой с товарами
- Фото первого товара как обложка (из WB CDN, если текст влезает в caption)
- Ссылки вшиты в текст (сохраняются при пересылке)
- Кнопки "WB/Ozon/ЯМ ↗" + "✘ не тот товар" для каждого товара
- Очередь запросов с семафором (1 одновременно) + статус "в очереди"
- Валидация BOT_TOKEN и OPENAI_API_KEY при старте
- **Результат: рабочий бот на Railway, Fast Path ~5 сек, Full Path ~30 сек**

### Деплой
- Railway, авто-деплой из GitHub при push
- Docker: python:3.11-slim + ffmpeg, non-root user
- Cookies.txt в репозитории (приватный репо)

## Нагрузка и стабильность

- Семафор: 1 одновременный Full Path запрос (защита от OpenAI rate limit)
- OCR retry: при сбое батча — повтор через 3 секунды (1 попытка)
- Очередь: пользователь видит "⏳ Ваш запрос в очереди (перед вами: N)..."
- Async: format_bot_response обёрнут в asyncio.to_thread (не блокирует event loop)
- Тестирование: 4 рилса подряд — все обработаны корректно, 14/14, 12/12, 20/20 товаров

## Известные ограничения

1. **Instagram rate limit** — instaloader блокируется при массовых запросах
2. **OCR точность 97%** — GPT-4o иногда добавляет лишние цифры к артикулам (832274563 вместо 83227463)
3. **Покрытие 96%** — на экстремальных рилсах (79 товаров) теряет 2-3 из-за OCR ошибок в цифрах
4. **WB CDN** — не работает для новых товаров (vol > 3701), но ссылки WB работают
5. **Cookies** — нужен файл cookies.txt с активной сессией Instagram
6. **Storage** — MemoryStorage в памяти, счётчик сбрасывается при рестарте
7. **ЯМ ссылки** — `/card/x/{id}` работает для большинства, если продавец удалил предложение — "нет в продаже" (артикул в тексте для ручного поиска)
8. **Параллельность** — 1 Full Path одновременно (OpenAI rate limit), остальные в очереди

## Юридические решения

- Видео: временное скачивание, удаление в finally-блоке (<60 сек)
- Инициатор: пользователь сам шерит ссылку
- Хранение: только артикулы и ссылки, не контент
- ToS: обработка по запросу, TTL <60 сек

## Код-ревью (8 апреля)

Проведено глубокое ревью через 5 параллельных агентов:
1. **Regression Hunter** — найден обход WantaService в bot.py, потеря контекста в carousel OCR
2. **Minimalist Critic** — 23% мёртвого кода (exceptions не используются, OCRResolver — заглушка)
3. **Bug Hunter** — 3 критичных бага (hardcoded .mp4, blocking event loop, нет валидации API key) — ПОЧИНЕНЫ
4. **Standards Enforcer** — общая оценка C+, главная боль: тесты (D), error handling (C)
5. **Elegance Architect** — предложил схлопнуть 30 файлов → 14, но решено сохранить директории для расширяемости

Решение: архитектуру сохраняем, баги починены, остальное доделаем по мере роста продукта.

## A/B тест OCR промптов (8 апреля, сессия 2)

### Методология
- 23 рилса, оба промпта на одних и тех же URL
- HEAD-верификация артикулов (оказалась бесполезна: WB=498, Ozon=403 — антибот)
- Ручная проверка 66 ссылок из самого сложного рилса (DVx_9gkNSgH, 79 товаров)

### Результаты
| Метрика | Промпт A (старый) | Промпт B (новый) |
|---------|-------------------|-------------------|
| Всего товаров (23 рилса) | 247 | 283 (+15%) |
| Фантомные артикулы | 8 | 0 |
| Точность | ~92% | 97% |
| DVx_9gkNSgH | 23 товара | 76 товаров |

### Ключевые находки
1. GPT-4o копировал `"123456789"` из примера в промпте → заменили на `"..."`
2. HEAD-верификация не работает для WB/Ozon — серверные запросы блокируются
3. Дедупликация по размеру файла убивала нужные кадры — товарные карточки слишком похожи по весу
4. Адаптивный подход: короткие видео (≤15с) без дедупа, длинные (>15с) с дедупом
